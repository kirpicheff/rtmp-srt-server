# План улучшений SRT-сервера

## 1. Активация логирования статистики входящего потока

### Что изменить:
Раскомментировать и доработать строки 299-300 в функции `handleConnection` для логирования статистики:
```go
elapsed := time.Since(startTime).Seconds()
if elapsed > 0 {
    bitrate := float64(totalBytes) * 8 / elapsed / 1000 // kbps
    avgPacketSize := float64(totalBytes) / float64(packetCount)
    log.Printf("[SRT] Input stats - Packets: %d, Bytes: %d, Bitrate: %.1f kbps, AvgPacketSize: %.1f bytes",
        packetCount, totalBytes, bitrate, avgPacketSize)
}
```

### Эффект:
Позволит отслеживать качество входящего потока, выявлять проблемы с битрейтом и размером пакетов в реальном времени.

## 2. Улучшенная обработка ошибок в RTMP output

### Что изменить:
В функции `processRTMPStream` добавить более детальную обработку отсутствия PTS:
```go
if data.PES.Header.OptionalHeader == nil || data.PES.Header.OptionalHeader.PTS == nil {
    // Вместо простого continue, добавить счетчик пропущенных пакетов
    missingPTSPackets++
    if missingPTSPackets > 100 { // Лимит пропущенных пакетов
        return fmt.Errorf("too many packets without PTS, stream may be corrupted")
    }
    continue
}
```

### Эффект:
Повысит устойчивость к поврежденным потокам и позволит быстрее обнаруживать проблемы с источником.

## 3. Адаптивная буферизация

### Что изменить:
Сделать размер буфера выходов настраиваемым в конфигурации:
```go
ch := make(chan []byte, cfg.OutputBufferSize) // Вместо фиксированного 1000
if strings.HasPrefix(outputURL, "rtmp://") {
    ch = make(chan []byte, cfg.RTMPOutputBufferSize) // Вместо фиксированного 5000
}
```

### Эффект:
Позволит оптимизировать использование памяти под конкретные сценарии использования и уменьшить вероятность переполнения буфера.

## 4. Добавление таймаутов при чтении

### Что изменить:
В функции `handleConnection` установить таймаут для чтения:
```go
conn.SetReadDeadline(time.Now().Add(30 * time.Second)) // Добавить перед чтением
n, err := conn.Read(buffer)
```

### Эффект:
Предотвратит зависание при проблемах с сетью и позволит быстрее обнаруживать обрывы соединения.

### Важное пояснение:
Таймаут чтения **не создает задержку в потоке**. Наоборот, он защищает от зависания при проблемах с сетью:
- При нормальной работе данные приходят сразу, и `Read` возвращает их без ожидания
- Таймаут срабатывает только при отсутствии данных в течение 30 секунд
- Это защитный механизм, который срабатывает только при проблемах с сетью

## 5. Улучшенное управление горутинами

### Что изменить:
Добавить контекст с таймаутом для всех горутин и механизм их корректного завершения:
```go
// При создании горутины
ctx, cancel := context.WithTimeout(parentCtx, 30*time.Second)
defer cancel()
```

### Эффект:
Предотвратит утечки горутин и обеспечит корректное завершение всех процессов при остановке сервера.

## 6. Обработка частичных чтений

### Что изменить:
В функции `handleConnection` добавить цикл для обеспечения полного чтения:
```go
var totalRead int
for totalRead < len(buffer) {
    n, err := conn.Read(buffer[totalRead:])
    if err != nil {
        return err
    }
    totalRead += n
}
```

### Эффект:
Обеспечит корректное чтение данных даже при фрагментации сетевых пакетов.

## 7. Ограничение на количество повторных подключений

### Что изменить:
В функциях `handleSRTOutput` и `handleRTMPOutput` добавить счетчик попыток:
```go
reconnectAttempts := 0
maxReconnectAttempts := cfg.MaxReconnectAttempts // Новый параметр в конфигурации

for {
    // ... логика подключения
    if err != nil {
        reconnectAttempts++
        if reconnectAttempts > maxReconnectAttempts {
            return fmt.Errorf("max reconnect attempts reached")
        }
        // ... остальная логика
    } else {
        reconnectAttempts = 0 // Сбросить счетчик при успешном подключении
    }
}
```

### Эффект:
Предотвратит бесконечные попытки переподключения при постоянных ошибках и позволит быстрее обнаруживать критические проблемы.

## Советы по длительному тестированию ретрансляции

### 1. Тестирование на протяжении 24+ часов
- Запустите стрим на 24-48 часов с мониторингом логов
- Отслеживайте утечки памяти и рост количества горутин
- Проверяйте стабильность битрейта и задержки

### 2. Стресс-тестирование
- Используйте несколько одновременных входов/выходов
- Проведите тесты с высоким битрейтом (10+ Mbps)
- Проверьте поведение при сетевых задержках и пакетных потерях

### 3. Мониторинг системных ресурсов
- Отслеживайте использование CPU/RAM в течение длительного времени
- Проверяйте дескрипторы файлов и сетевых соединений
- Мониторьте логи на предмет повторяющихся ошибок

### 4. Тестирование восстановления после ошибок
- Искусственно разрывайте соединения во время стрима
- Перезапускайте выходные сервисы (RTMP/SRT сервера)
- Проверяйте корректность восстановления потока

### 5. Логирование и метрики
- Включите детальное логирование всех операций
- Добавьте метрики для мониторинга (Prometheus/Graphana)
- Создайте дашборды для визуализации состояния сервера

### 6. Тестирование с реальными устройствами
- Используйте реальные энкодеры (OBS, Wirecast, аппаратные энкодеры)
- Проверьте совместимость с различными CDN платформами
- Протестируйте с мобильных устройств как источников
